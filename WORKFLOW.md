# 智潮 / SIGNAL TIDE 执行工作流

这是一份 **执行用工作流**。

目的不是介绍项目，而是确保每天更新时：

- 抓取正确
- 写稿正确
- 多语言正确
- 页面不出错
- 归档不出错
- GitHub Pages 正常发布

如果本文件和你“印象中的流程”冲突，**以本文件为准**。

---

# 1. 产品定位

智潮 / SIGNAL TIDE 是：

**由 openclaw 抓取、筛选、编辑、翻译并发布的全球 AI 日报。**

它不是：

- RSS 直出站
- 纯转载站
- 只会列标题的聚合器
- 依赖后端服务的动态新闻系统

它是：

- 一个纯静态日报站
- 内容每天生成一版
- 页面展示像报纸 / 日报，而不是资讯流或控制台

---

# 2. 自动更新时间

**正式更新时间：每天 UTC 02:00**

理由：

- 能尽量覆盖欧美上一整天的重要 AI 新闻
- 对全球用户更合理
- 对亚洲用户也仍然属于白天可读时间
- 更符合“全球日报”而不是“本地晨报”

不要擅自改成更早的时间，除非用户明确要求。

# 4. 每日执行总流程

每天 UTC 02:00 自动任务必须做这几步：

1. 阅读本文件 `WORKFLOW.md`
2. 抓取候选新闻源
3. 更新 `raw-feeds.json`
4. 选题并生成当天稿件
5. 补全全部语言
6. 写入 `issues.json`
7. 刷新 `rss.xml`
8. 刷新当天归档文件
9. 同步本地预览目录
10. 检查页面是否正常
11. 推送到 GitHub
12. 发一条简短 Telegram 通知

任何一步失败，都不要发布空版面。

# 5. 新闻抓取规则

## 5.1 候选源来源

候选源要覆盖这些面向：

- 官方 / 一手发布
- 科技媒体
- 研究博客 / 论文相关
- 开源 / 开发者生态
- 政策 / 治理 / 安全

## 5.2 抓取命令

进入目录：

```bash
cd /root/.openclaw/workspace/ai-news-pages-demo
```

抓取候选：

```bash
npm run fetch:feeds
```

输出文件：

```text
docs/data/raw-feeds.json
```

## 5.3 候选池原则

- 候选池是线索，不是成稿
- 不要把候选标题直接当页面文案
- 候选越多越好，但成稿必须克制筛选

---

# 6. 选题原则

每天的版面必须有明确层级。

## 6.1 首页当前栏目

首页当前结构是：

- 头版
- 今日要闻
- 趋势
- 观察
- 开源项目推荐

## 6.2 选题要求

### 头版
- 只能有 1 条
- 必须是当天最值得当封面的新闻
- 标题必须短，不能太拖
- 要像报纸头条，不像博客标题

### 今日要闻
- 放影响力第二梯队
- 更偏明确事件和重要更新
- 不要塞太多公司同类消息

### 趋势
- 放更短、更密的条目
- 适合产品动态、行业走向、应用趋势
- 要紧凑，不要拖太长

### 观察
- 放研究、方法、结构变化、长期值得看的内容
- 不允许空白
- 如果偏空，优先补文章，不要先靠 CSS 硬撑

### 开源项目推荐
- 保持紧凑
- 不抢头版和正文区的版面

## 6.3 多样性要求

当天版面尽量覆盖：

- 公司与产品
- 模型 / 安全
- 政策治理
- 开源工具 / 开发者生态
- 研究与长期影响

不要让同一家公司占据大多数版面。

# 7. 写稿规范

每篇文章至少要有这些字段：

- `id`
- `section`
- `category`
- `headline`
- `deck`
- `kicker`
- `byline`
- `publishedAt`
- `updatedAt`
- `readingTime`
- `tags`
- `summary`
- `bullets`
- `significance`
- `sources`

## 7.1 标题规范

- 尽量短
- 头版标题尤其不能太长
- 不直接照搬原标题
- 不要像 SEO 标题
- 不要像新闻推送通知标题

## 7.2 deck 规范

- 解释“为什么重要”
- 比标题多一层信息
- 不要只是标题改写

## 7.3 summary 规范

- 一般 2 段
- 必要时 3 段
- 不要堆空话
- 要像日报编辑整理后的正文提要

## 7.4 significance 规范

- 必须说明这条新闻更长期的意义
- 可以偏判断，但不能编造事实

## 7.5 sources 规范

- 必须保留来源链接
- 尽量优先放一手来源
- 媒体来源可以作为补充

- `zh-CN`

- `en`
- `ja`
- `ko`

- `fr`
- `es`

写入位置：

```text

```

文章相关内容必须在 `issues.json` 里：

- `headline`
- `deck`
- `kicker`
- `category`
- `summary`
- `significance`
- `sources`

不要把正文多语言写死在前端 JS 里。

## 8.2 UI 文案规则

UI 文案在：

```text
docs/locales/*.json
```

例如：

- 站点标题
- 返回按钮
- 来源
- 阅读时间
- 关于
- 归档
- 今日要闻
- 趋势
- 观察

## 8.3 站名规则（固定）

站名必须遵守：

- 简体中文：`智潮`
- 繁体中文：`智潮`
- 其他所有语言：`SIGNAL TIDE`

不要改成别的英文变体。
不要让 `issues.json` 里的站点对象覆盖掉这条 UI 规则。

# 9. `issues.json` 数据格式规范

## 9.1 必须兼容两种数据形态

页面读取时，必须兼容：

### 形态 A：普通值

```json
{
  "headline": "标题",
  "deck": "导语",
  "summary": ["第一段", "第二段"]
}
```

### 形态 B：多语言对象值

```json
{
  "headline": {
    "zh-CN": "标题",
    "en": "Headline"
  },
  "summary": [
    {
      "zh-CN": "第一段",
      "en": "Paragraph 1"
    }
  ]
}
```

## 9.2 必须兼容的字段

前端读取时，必须正确展开这些字段，不能出现漏修：

- `headline`
- `deck`
- `kicker`
- `category`
- `byline`
- `significance`
- `summary[]`
- `bullets[]`
- `sources[].title`
- `sources[].publication`

# 10. 明确禁止出现的错误

## 10.1 页面不能出现 `[object Object]`

如果页面出现：

```text
[object Object]
```

说明前端没有把多语言对象展开。

这是严重错误。

一旦出现，优先检查：

- 头版标题
- 今日要闻正文
- 详情页正文
- 来源栏
- 观察区

## 10.2 不能发布空版面

不允许：

- 头版空白
- 今日要闻空白
- 趋势空白
- 观察空白
- 归档页跳错

如果内容不足：

- 先补稿
- 再调布局
- 不要先靠 CSS 撑高度掩盖问题

## 10.3 不要 silently fallback 到错误日期

归档页规则：

- 如果用户输入有归档的日期 → 打开那一天
- 如果用户输入没有归档的日期 → 跳转 `404.html`
- 提示：`没有该档案`

不能偷偷回退到昨天或别的日期。

## 10.4 不要让 GitHub Actions 和本机 cron 冲突

当前自动更新以 **本机 OpenClaw cron** 为准。

不要重新启用旧的 GitHub Actions 夜间自动发布逻辑，除非用户明确要求。

# 11. 首页与归档页要求

## 首页

- 读：`docs/data/issues.json`
- 展示当天版面

## 归档页

- 默认打开：昨天的报纸
- 如果 URL 带 `?date=YYYY-MM-DD`，则尝试打开该日期
- 右上角使用：
  - 日期输入框 `#archiveDatePicker`
  - 右箭头按钮 `#archiveGo`
- 没有归档时：跳到 `404.html`

## 文章详情页

- 如果来自归档页，返回要回到对应日期的归档页
- 不能直接把用户丢回当天首页
